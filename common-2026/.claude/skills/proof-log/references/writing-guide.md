# proof-log の書き方ガイド

`SKILL.md` の「ステップ6: 質的ログを書く」を実行するときの指針。テンプレート (`template.md`) と合わせて読む。

## 書く目的を見失わない

proof-log は **「次に同じことをするときの claude code がどんなツールを持っていれば速くなったか」を後から判断するための一次データ**。だから:

- 「うまくいった話」より **「詰まった話」** が価値がある
- 「思いついたアイデア」より **「思いつくまでの探索」** が価値がある
- 「最終的な解法」より **「採らなかった代替案」** が価値がある

最終的な解法は git log と最終ファイル（`Common2026/*.lean`）に残っている。proof-log でそれを再説明するのは冗長。

## 書く / 書かない

| 書く | 書かない（理由） |
|---|---|
| 実際に打った grep のクエリ | grep の存在自体（自明） |
| 空振りした検索語と試行回数 | 自分の「体感ツール数」（誤差大） |
| Mathlib に無かった補題と回避策 | 自分の「体感所要時間」（誤差大） |
| ハマった LSP / build / 型エラー | エラーの一般論 |
| `private` / `lemma` / `instance` の可視性で躓いた具体例 | 「Lean 4 は難しい」 |
| 「気づくのに時間がかかった」数学的観察 | 教科書に載っている定理の説明 |
| 採らなかった分割 / 別経路の理由 | 採用した解法の自慢 |

## 数値は metrics.md に任せる

proof-log の本文に**自分で数えた数値を書かない**。次のような表現はすべて避ける:

- 「ツールコール約 30〜40 回」
- 「目視カウント、誤差 ±2」
- 「約 1 時間〜2 時間相当の集中作業」
- 「`lake env lean` を約 12 回」

代わりに:

- 「定量データは [metrics.md](metrics/<slug>.metrics.md) を参照」とだけ書く
- どうしても本文中で言及する必要がある場合（例: 工程比率の議論）は metrics.md の数字を直接引く

例外として「**体感比率**」（"50% は補題探索に費やした感がある"）は主観で書いて構わない — メトリクスから比率は機械的には出ない（ツール種別ごとの所要時間配分は分かるが、ユーザーの集中の主観配分はわからない）。ただし「これは主観」と明記する。

## 「無かった」の書き方

Mathlib に存在しなかった補題は **章を立てて書く**。書き方:

```markdown
- **`integral_abs_cos`** — `integral.*abs_cos`、`abs_cos.*integral`、`integral_norm_cos` の 3 つで空振り。
  Mathlib にあるべき形だが見つからず、3 区間分割で手書き（70 行、最終ファイルの約 25%）。
  教訓: 型ベース検索があれば「`∫₀^{2π} |cos x| dx = ?` の形で問い合わせる」と即時に「無い」が確定したはず。
```

ポイント:

- **試した検索語を全部書く**（後でツール開発側が「この検索語の集合を eval にする」可能のため）
- **コストを定量化**（行数 / 全体に占める比率）
- **教訓**を「将来のツールがこれをどう解決すべきか」の形で書く

## 「失敗」の書き方

時系列で語らない。原因と教訓を先に出す。

❌ 悪い例:

> grep を打ったが見つからなかった。次に別の grep を打ったが、これも見つからなかった。3 回目でようやく見つけた。

⭕ 良い例:

> **症状**: `monotone_of_deriv_nonneg` を探して `Mathlib/Analysis/Calculus/MeanValue.lean` 単体に grep して空振り。
> **原因**: 主要補題は `Mathlib/Analysis/Calculus/Deriv/MeanValue.lean`（階層が一つ深い別ファイル）にある。
> **抜け方**: ディレクトリ全体を `grep -rl` で再検索。
> **教訓**: ピンポイント grep は脆い。Mathlib のモジュール分割が更新されると過去の知識が無効になる。最初から階層全体を検索 → ヒットしたファイル名を読む、の 2 段階構成にすべき。

## 「ボトルネックではなかったもの」を書く理由

これを書かないと「Lean 形式化はあらゆる工程が大変」というぼんやりした印象が残る。**逆に「ここはツールが要らない」を明示することがツール開発の優先順位付けに直結する**。

例:

- 「数学のアイデアは要らなかった（既知の系の組合せ）」 → 「アイデア生成支援」より「補題検索支援」が優先
- 「型チェックでは詰まらなかった」 → 「型推論補助」は優先度低
- 「コンテキスト長は十分だった」 → 「コンテキスト圧縮ツール」は不要

## 文体

- 日本語、です・ます調と常体は混在してよいが**1ファイル内では揃える**（既存サンプルは常体）
- 表は積極的に使う（行数比較、検索クエリ、優先度マトリクス）
- 引用ブロックで grep コマンドや Lean コードを残す
- 絵文字は使わない（既存サンプルに合わせる）
