# carryon: 前セッションのハンドオフから再開

`/handoff` で書かれた `.claude/handoff.md` を読み、状態を内部で復元してから作業を再開する。

## やること

1. **`.claude/handoff.md` を Read**
   - ファイルが無い場合: ユーザーに「ハンドオフファイルが無い。今セッションは新規？」と確認して終了
2. **Task list を復元** — handoff.md の Tasks セクションに pending/in_progress 項目があれば、`TaskCreate` で同じ subject + description で作り直す (status は元と同じ)
3. **状態確認のため軽く検証** (並列で OK):
   - `git status` (ハンドオフ時点と一致するか)
   - `git log --oneline -3` (新たなコミットが入っていないか)
   - 「Files to read first」のうち上位 1〜2 個を Read
4. **再開準備が整ったことをユーザーに報告し、次に何をするかを宣言**:
   - ハンドオフ日時
   - 復元したタスク数
   - 「Next step」をそのまま提示 — これがそのまま「これから着手する作業」の宣言になる
   - もし状態に不整合があれば指摘 (例: ハンドオフ後に他者がコミットしている、ファイルが消えている)
5. **宣言した作業にそのまま着手する** — 確認を待たない。ただし状態に不整合があるとき、または Next step が破壊的 / 不可逆な操作 (push, force-push, 大規模リファクタ等) を含むときは、例外的に確認を取ってから進める
6. **オーケストレーターとして連続自走する** — 着手後は 1 ステップで止まらず、下記「デフォルト挙動」に従って停止条件に達するまで複数セッション分を進める

## デフォルト挙動: オーケストレーターで連続自走

特別な指示 (「1 手だけ」「ここまでで止めて」等) がなければ、再開後は **オーケストレーターパターンで複数セッション分を連続して進める**。1 つの Next step を終えても都度ユーザーに伺いを立てず、handoff の残タスク / プランに沿って次の手へ自律的に繋ぐ。実装・調査・docs 編集は subagent に dispatch し、自分は計画・検証・統合・commit/push を回す (CLAUDE.md「Parallel orchestration」「Orchestrator role 規律」準拠)。

### 停止条件 (いずれかに達したら自走を止めてユーザーに返す)

1. **タスク全体が完了** — handoff のスコープ / プランの目標を満たし、残タスクが無い。完了サマリを報告して終了。
2. **ユーザーの判断・意思決定が必要** — 方針分岐 / トレードオフ / 撤退ライン該当 / 不可逆操作の承認など、自分の裁量で決めるべきでない論点に当たった。`AskUserQuestion` 等で判断を仰ぐ。
3. **コンテキスト圧迫で判断精度の低下が懸念される** — 会話が長大化し、自分の判断の正確さが落ちるリスクを感じたら、無理に続けず `handoff` スキルで状態を書き出してリフレッシュを促す (次セッションが `/carryon` で拾える形にする)。

停止条件 1〜3 のいずれにも当たらない限り、自走を継続する。停止時はどの条件で止まったかを明示する。

## 報告フォーマット (短く)

```
ハンドオフ from <日時> 復元。
- タスク #N, #N+1 を再作成
- 状態: <整合 / 不整合の指摘>
- 次の一手: <handoff.md の Next step を 1〜2 行に圧縮> — これから着手する。
```

そのまま続けて最初の tool call に入る。

## 原則

- **handoff.md の内容を全部復唱しない**。要点だけ。詳細はファイルにある
- **宣言してから動く、ただし許可は求めない**。Next step がそのまま着手予告になる。ユーザーは宣言を読んで違うと思えば止められる
- **状態に不整合があるときは止まる**: ハンドオフ書いた後にユーザーが手で編集 / コミット / branch 切り替えをしているかもしれない。`git log` を見て新規コミットがあれば必ず指摘し、その場合は確認を取る
